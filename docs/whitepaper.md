# Validating agentic-AI defenses in financial services: a reproducible red-team range

*A short paper accompanying [FinAgent-RedRange](https://github.com/emmanuelgjr/finagent-redrange).
Every number below is regenerated by one command; the repository is the reproducible artifact.*

## Abstract

Agentic AI moves the attack surface in banking from "bad text" to **real actions on customer
accounts** — transfers, KYC lookups, support operations. Point-in-time penetration testing does not
keep pace with continuous model and prompt change, and a flashy proof-of-concept exploit is not the
same as evidence that a defense works. FinAgent-RedRange is a self-contained, offline, deterministic
range that develops POC exploits against a representative mock banking agent and **proves that a
specific guardrail closes each one** — end to end, from exploit through a passing regression test —
with every finding mapped to OWASP LLM Top 10, the OWASP Agentic schemes, MITRE ATLAS, and NIST AI
RMF. The deliverable is *evidence that defenses hold*, not an attack toolkit.

## Why financial-services agents need their own threat model

A retail-banking agent is not a chatbot. It runs a plan→act→observe loop over permission-checked
tools, retrieves from a knowledge corpus, and can move money. That combination raises three stakes a
bank cannot wave away: **customer PII and cross-account isolation**, **irreversible financial
actions**, and **auditability** for regulators. And the system is never static — the model, the
system prompt, the tools, and the retrieved content all change continuously. A control that held
last quarter can silently regress. What a risk function actually needs is not a one-off report but a
**continuous assurance loop**: for each known attack, prove the exploit is real and prove the
mitigation still blocks it, on every change, mapped to the frameworks the team already audits
against.

## Method: a POC-to-validation range

The organizing invariant is simple and strict: **no exploit without its fix.** A finding is only
"done" when the control that neutralizes it is proven by a regression test that runs the *same
attack* twice — once with controls off (it must land) and once with controls on (it must be blocked).
If a control cannot flip its scenario's oracle, the scenario is incomplete. This turns ad-hoc
red-teaming into repeatable, auditable evidence.

Three design choices keep that evidence honest:

- **Black/grey-box discipline.** The attacker only touches the agent's public `respond()` surface and
  judges success from the transcript — the same position a real adversary occupies.
- **A deterministic offline simulator, clearly labeled.** The default `EchoClient` is a *compliant-
  agent simulator*: it reproduces the failure mode so the controls have something concrete to
  neutralize. The offline pass therefore proves the **harness and the controls** work in CI — it is a
  regression fixture, not a claim that a frontier model is exploitable. That distinction is stated
  everywhere it matters.
- **Framework mapping by construction.** Findings carry structured OWASP/ATLAS/NIST identifiers as
  data, verified against the published standards, so they drop straight into governance workflows —
  and blank where no honest mapping exists, rather than forcing a code.

## The range

The target is a mock retail-banking agent with five permission-checked tools (balance, transfer,
KYC, transactions, support ticket), RAG over a policy corpus, and four toggleable guardrail layers
(input, retrieval, action, output). Nine attack classes exercise the whole pipeline — the **full
OWASP LLM Top 10** plus a **multimodal input surface**:

| Attack (OWASP) | Validating control |
|---|---|
| Indirect prompt injection → cross-account PII (LLM01/02) | Output PII filter + retrieval provenance |
| Data poisoning → fabricated policy (LLM04/09) | Source allowlist + corpus integrity hash |
| Excessive agency → autonomous transfer (LLM06) | Action-authorization gate (human-in-the-loop) |
| System-prompt leakage (LLM07) | Output canary / verbatim-span detector |
| Unsafe output handling → phishing link (LLM05) | Output link/markup sanitiser |
| Vector/embedding weakness → cross-tenant leak (LLM08) | Access-scoped retrieval |
| Unbounded consumption → cost exhaustion (LLM10) | Per-request tool-call budget |
| Supply chain → malicious third-party tool (LLM03) | Verified-publisher allowlist |
| Multimodal injection → image-borne instruction (LLM01) | Multimodal input guardrail (OCR as data) |

An **autonomous attacker** composes seed payloads with transforms until an oracle fires, behind a
pluggable planner: a deterministic sweep (offline, CI-friendly) or an adaptive **LLM-driven planner**
that reasons about the next probe from the feedback of prior attempts. Its seeds can be drawn from a
curated real-world **incident corpus** — technique, real-world prioritization, and provenance only;
every payload stays a synthetic probe against the mock agent.

## Results

The headline is the scorecard: **every scenario is exploited with controls off and blocked with
controls on**, and the mean AIRQ risk heuristic shifts High→Medium when controls engage. (AIRQ is an
*analyst heuristic for ordering work*, not a calibrated metric; the paper does not quote it as
residual risk.) A CI regression suite re-proves the controls-on invariant on every change across
Python 3.11 and 3.12.

Two results are worth calling out because they answer the two hardest questions a reviewer asks.

**"Does a real model actually fall for this?"** Yes — and the range demonstrates the full
exploit→block transition against a live model, not just the simulator. Against `claude-haiku-4-5`,
the **supply-chain** scenario landed with controls off: steered by a retrieved note, the model
invoked an untrusted "recovery partner" plugin (`account_recovery_helper`) that exfiltrated the
session's data — because *invoking a helper to unlock a customer does not read as malicious*, so
there is no harmful request for the model to refuse. With the verified-publisher control on, the
unverified tool is neither exposed nor callable, so the same model, given the same note, simply helps
within policy. The *control* — not the model's judgment — flips the outcome. The other eight
scenarios were natively refused by the model controls-off, which is the expected frontier-model
result and itself the point: **native resistance is probabilistic and model-dependent; the guardrail
is the deterministic guarantee.**

**"Can you detect what you mitigate?"** Each validated control is exported as a portable **Sigma**
detection rule, and a labeled-replay harness measures its precision against the range's own evidence:
every rule fires on its scenario's exploited transcript and stays silent on all others — **9 true
positives, 0 false positives** over the labeled corpus. The range also emits a SARIF findings run, a
GSN control-effectiveness assurance case (every claim bound to a passing test and a transcript hash),
a regulatory crosswalk (NIST/ISO 42001/EU AI Act, with declared-vs-interpretive provenance), and a
MITRE ATLAS coverage layer — the handouts a detection engineer, an architect, and a GRC lead can each
pick up unchanged.

## Honest limitations

Credibility comes from stating the edges. The pattern-based guardrails are **heuristics** — narrow by
design, but bypassable by paraphrase or novel encoding; they are one layer, not a guarantee. The
offline evidence proves the harness and controls, not model exploitability. Real-model runs are
non-deterministic and, so far, show one demonstrated exploit→block; a larger real-model evaluation
(landing-rate statistics) is future work. The target is single-agent, so multi-agent failure modes
(inter-agent communication, cascading failures, rogue agents) are not yet modeled — the natural next
frontier. And OWASP LLM08 is modeled with audience-scoped retrieval rather than a production embedding
store. None of these are hidden; each is noted in the code and roadmap.

## From range to continuous assurance

The value is not the nine payloads — those are documented technique categories. It is the **loop**: a
reproducible, framework-mapped, CI-gated pipeline where every finding ships with its proof-of-fix, a
real model has been shown getting exploited and then blocked by the named control, and the results
are handed to security teams in the formats they already run. Wire the regression suite into CI, gate
GA agent releases on it, seed the attacker from your incident history, and a bank has what
point-in-time testing cannot give it: continuous, auditable evidence that its agent's defenses hold —
and keep holding — as the model underneath it changes.
